# Curator Guide — `knitting-stitches/`

This directory is **not application code**. It is the curation system for the Knitr semantic vocabulary: every knitting stitch we recognize, plus every visual symbol that real-world legends and charts use to denote each stitch.

The curated output is exported into `packages/schema/src/generated/` and consumed by the app's legend matcher, symbol picker, and chart renderer.

## Purpose

Knitters writing patterns use wildly inconsistent abbreviations and an even wider range of chart symbols. The curator is what lets the app handle that:

- **Search any abbreviation or legend text** ("k2tog", "k2t", "knit two together", "S2KP2", "sl1-k2tog-psso", etc.) and resolve it to a canonical stitch.
- **Show every visual variant** a knitter might encounter for that stitch (CYC, JIS, German, StitchMastery, custom).
  - **Note:** the `house-symbols/` recipe-based system is **deprecated** — its mappings are no longer included in the curated library. The source files are kept in `house-symbols/` for reference but not built or surfaced in the curator UI.
- **Stay faithful** to the distinctions real publishers make, even when those distinctions are subtle (e.g. `cdd` vs `cdd_tbl` vs `s2kp` all share the peaked-triangle visual but are mechanically different stitches).

## Coverage philosophy

**Coverage over efficiency.** When in doubt, split:

- Cover JIS, CYC, StitchMastery, German, and any other knitting standard we encounter.
- Do **not** merge stitches that real publishers treat as distinct, even if their chart symbols look identical.
- Do **not** merge symbol variants either — if two SVGs are visually different but semantically equivalent (e.g., a CYC peaked triangle and a JIS peaked triangle for `cdd`), keep both. A knitter looking at their pattern needs to recognize *their* publisher's symbol.
- The same visual *can* appear under multiple stitches when publishers reuse shapes and disambiguate via legend text. See **Data model** below for the duplicate-`symbol_id` pattern.

## Edit ownership

Strict separation between agent-writable and human-writable files:

- **Agent edits** (this includes Claude Code or any other automated tool):
  - `stitches/*.json` — canonical stitch definitions (ids, abbreviations, semantic descriptions, stitch math).
  - `symbol-mapping.json` — the candidate pool the curator UI reads from. Add new entries (including duplicate-`symbol_id`-with-suffix entries for shared visuals) here so they appear as candidates in the UI for the user to accept or reject.
  - `CURATOR.md` / `AUDIT.md` / other docs.

- **User edits** (manual, via the curator UI):
  - `curated-symbols-current.json` — produced as the curator's export. The user accepts/rejects candidates in `curate.html` and clicks Export.

- **The agent must NOT write to `curated-symbols-current.json` directly.** Doing so:
  1. Bypasses the user's manual review step (the whole point of curation).
  2. Risks being silently overwritten the next time the user exports from the curator UI (their localStorage is the source of truth for in-progress curation state).
  3. Creates a divergence the curator UI can't see — the entry won't show up as a candidate, only in the exported result.

  Exception: large coordinated migrations explicitly approved by the user, where the user understands they're skipping the review step. Default to symbol-mapping.json.

## Required agent consultations

Per `CLAUDE.md`, when working in `knitting-stitches/` you **must** consult:

- **`knitting-notation-parser`** — for chart conventions, legend conventions, RS/WS inference, repeat detection, stitch math.
- **`knitting-expert`** — for stitch semantics, real-world publisher behavior, and judgment calls on whether two stitches are genuinely distinct.

Don't guess at knitting domain questions. The split between `cdd`, `cdd_tbl`, `sk2p`, `s2kp` etc. is the kind of thing that needs a knitting opinion, not a code opinion.

## Workflow

```
stitches/*.json                  ← canonical stitches (hand-edited)
       │
       ▼
map_symbols.py                     ← auto-maps SVGs from sources to stitches
       │
       ▼
symbol-mapping.json                ← candidate pool (agent-writable; auto-generated by map_symbols.py + agent additions for shared-visual duplicates)
       │
       ▼
curate.html (browser)              ← review, accept/reject each candidate
       │
       ▼
curated-symbols-current.json               ← user's curator export (do NOT hand-edit — see "Edit ownership" above)
       │
       ▼
build-app-symbol-library.ts        ← exports to packages/schema
       │
       ▼
packages/schema/src/generated/
   ├── app-symbol-library.{ts,js,d.ts}
   └── svg/<flattened-symbol-id>.svg
```

Run the build after every curation change:

```bash
pnpm exec tsx knitting-stitches/build-app-symbol-library.ts
```

## Data model

### Stitches are the source of truth

Each stitch has an `id` (e.g., `cdd`, `k2tog`), `surface_forms` (abbreviations and alternate names that legend search matches against), `stitch_manipulation` (semantic stitch math), and `execution` (steps for the knitter). See `schema.ts` for the full type and `stitches/decreases.json` for examples.

### One symbol → one stitch (in the export)

The build script keys `symbolsById` by `symbol_id`, so two stitches cannot share the same `symbol_id` — the second write silently overwrites the first.

### Sharing a visual across multiple stitches

When the **same SVG** legitimately represents **multiple stitches** (e.g., a peaked-triangle StitchMastery symbol that publishers use for `cdd`, `cdd_tbl`, *and* `s2kp`), use the **stitch-suffixed `symbol_id`** convention:

```
sm:sm-dash-30           → cdd       (primary, original symbol_id)
sm:sm-dash-30-cdd-tbl   → cdd_tbl   (duplicate, suffix = stitch id)
sm:sm-dash-30-s2kp      → s2kp      (duplicate, suffix = stitch id)
```

All three reference the same `svg_path`. The build copies the SVG once per `symbol_id` into separate output files. The app sees three symbol variants, each bound to its respective stitch.

This pattern is intentional: knitters look at their pattern's legend, see a peaked triangle next to "S2KP" (or "CDD" or "sl2-tbl-k1-p2sso"), and need to find that visual under whichever stitch their legend names. Because chart publishers reuse symbols, our library must too.

## Known limitations

- **`symbol-mapping.json` is auto-generated** by `map_symbols.py`. Manual edits to it (e.g., the duplicate-`symbol_id` entries described above) can be lost on a re-run. Until `map_symbols.py` learns about manual overrides, treat `curated-symbols-current.json` as the only durable source of truth and re-apply manual `symbol-mapping.json` edits after re-running the mapper.
- **The curator UI** (`curate.html`) is a single-file vanilla HTML app with localStorage persistence. It exports the full `curated-symbols-current.json` on demand. There is no server.
- **Adding a brand-new stitch** requires both an entry in `stitches/*.json` (so the registry knows about it) *and* curated symbols in `curated-symbols-current.json` (otherwise the build drops the stitch as having no variants).

## Quick reference

- New stitch → edit the appropriate `stitches/*.json` file. Mirror an existing stitch's shape; consult `schema.ts`.
- New symbol mapping → add the entry under the stitch in `curated-symbols-current.json`. Update `symbol_count` and `totals.accepted_symbols`.
- Shared visual across stitches → duplicate the `symbol_id` with a `-{stitch_id}` suffix and reuse the same `svg_path`.
- Always rebuild with `pnpm exec tsx knitting-stitches/build-app-symbol-library.ts` after curation changes.
